# システム要件定義書

## システム概要
FamilyTrip Planner（仮）は、幅広い年齢層の子供を連れた家族旅行に対応した旅程作成・共有プラットフォームです。子連れ旅行特有の悩み（トイレ、食事、安全性など）を解決するため、成功体験だけでなく「リアルな失敗情報や注意点」を可視化します。

1. **システムの目的**
   - **解決する課題**: 分散した子連れ旅行情報とリアルなネガティブ情報の欠如による、計画の手間や現地での失敗リスク。
   - **目標とする成果**: ユーザーが他者の成功・失敗体験を元に安全・安心な旅程を簡単に作成・実行でき、かつ次のユーザーへの情報提供サイクルが回ること。
   - **期待されるビジネスインパクト**: ニッチな課題解決による熱量の高い初期ユーザー獲得と、プレミアム機能などによる中長期的な収益化。

2. **主な機能**
   - **コア機能**: 旅程の検索・コピー、プランのカスタマイズ編集（ドラッグ＆ドロップ）、Google Mapへのエクスポート機能。
   - **差別化機能**: 失敗アラート機能（ネガティブ情報の可視化）、振り返り入力と連動した「思い出ムービー」の自動生成、子供向け「旅のしおり」のPDF生成機能。
   - **付加価値機能**: 今後のPhaseにおけるコミュニティ機能（役に立ったボタン等）。

3. **提供する価値**
   - **ユーザーにとっての直接的価値**: 計画時間の大幅短縮、失敗リスクの回避、旅行後の思い出の動画化。
   - **ビジネスにとっての価値**: 独自性の高いUGC（ユーザー生成コンテンツ）の蓄積と、エンゲージメントの高いコミュニティの形成。

4. **実現方法の概要**
   - **主要なテクノロジー要素**: Next.js (React) によるフロントエンド・バックエンド統合、Supabaseによるユーザー・旅程データ管理。
   - **重要なインテグレーションポイント**: GoogleマイマップへのKMLエクスポート、BGMやアニメーションを含んだ動画生成エンジンの組み込み。
   - **スケーラビリティへの考慮**: VercelのエッジネットワークとSupabaseのスケーラビリティを活かし、画像・動画などのリソースはCDNを介して高速配信。

## 機能要件

1. **アカウント・プロフィール管理**
   - Google認証 / SNS認証によるサインアップ・ログイン。
   - 家族構成（子供の年齢、性別）の設定。
2. **旅程検索・コピー**
   - エリア、子供の年齢、タグ（失敗しない、等）による公開プランの検索。
   - 選択した他者のプランを自アカウントの下書きとしてコピーする機能。
3. **旅程プランニング・編集**
   - コピーしたプランのスケジュール調整（ドラッグ＆ドロップによるスポットの入れ替えや時間変更）。
   - 各スポットに対する「失敗アラート（注意点）」アイコンの表示機能。
4. **外部連携（Google Map）**
   - 作成した旅程に含まれるスポット情報とメモ（タグ）を、Google Mapのリストやマイマップとしてエクスポートする機能。
5. **振り返り・投稿**
   - 旅行終了後、確定した旅程に対する評価と「一言コメント（成功/失敗）」の入力。
   - 失敗情報タグ（「トイレ汚い」「待ち時間長い」等）の選択入力。
   - 各スポットへの写真アップロード。
6. **思い出ムービー生成**
   - 入力された旅程データ、写真、コメントをもとにショートムービー（BGM・アニメーション付き）を自動生成。
   - 生成された動画のダウンロードおよびSNSシェア機能。
   - ムービー生成完了と同時にプラン全体を公開設定可能にする機能。

## 非機能要件
- **パフォーマンス**: 初期ユーザー1,000人規模を想定。検索結果の表示およびプラン編集画面の操作レスポンスは1秒以内を目指す。ドラッグ＆ドロップUIは60fpsで滑らかに動作すること。動画生成はバックグラウンドで非同期処理し、数分以内に完了通知を送る。
- **スケーラビリティ**: UGC（写真や動画）が増加することを見込み、ストレージおよびCDNの拡張性を担保する。
- **セキュリティ**: ユーザーのプライバシー（子供の写真など）を扱うため、非公開設定のデータは厳密なアクセスコントロールを行う（RLSの活用等）。
- **可用性**: VercelおよびSupabaseのマネージドサービスを活用し、99.9%のアップタイムを目指す。

## 技術スタック
以下のモダンな技術スタックを第一候補として選定します。これらは機能・非機能要件、スケーラビリティ、開発効率、コストを総合的に考慮した結果、本プロジェクトに最適です。

- **フロントエンド:** Next.js 14以降 (React)
  - 理由: SEO対策（Webからの検索流入重視）、リッチなUI（ドラッグ＆ドロップ等）の構築に適している。
- **言語:** TypeScript
  - 理由: 型安全によるバグの早期発見、保守性の向上。
- **バックエンド:** Next.js App Router
  - 理由: フロントエンドとバックエンドの境界をなくし、開発スピードを最大化できる。
- **データベース:** Supabase (PostgreSQL)
  - 理由: RLS (Row Level Security) による安全なデータアクセス、リアルタイム連携、Storage機能（画像・動画保存）がオールインワンで提供されており初期コスト・運用コストが低い。
- **認証:** Supabase Auth または Clerk
  - 理由: Next.jsとの親和性が高く、SNSログインの導入が極めて容易。（Supabase Authに寄せるとスタックがシンプルになるため第一候補）
- **ホスティング:** Vercel
  - 理由: Next.jsのフル機能をゼロコンフィグで引き出せ、エッジネットワークによる高速な配信が可能。
- **その他（動画生成）:** Remotion または FFmpegベースのLambda処理
  - 理由: 思い出ムービーの自動生成をReactコンポーネントベースで実現する場合、Remotionの相性が良い。

## 開発ロードマップ
- **Phase 1: MVP開発（約1.5〜2ヶ月）**
  - **目的**: コア価値（他人のプランコピーとGoogle Map連携、簡易振り返りと動画生成）の検証。
  - **内容**: 認証、プラン検索・コピー・編集、GoogleマイマップへのKMLエクスポート、振り返り入力、BGM・アニメーション付きのムービー生成機能、子供向け「旅のしおり」PDF生成機能。Webブラウザ（スマホ最適化含む）のみ。
- **Phase 2: コミュニティ機能強化・UIブラッシュアップ（リリース後1〜2ヶ月）**
  - **目的**: 投稿の質と量の向上、UGCの好循環の形成。
  - **内容**: 役に立ったボタン、ユーザーランク（スタンプ）の本格実装、動画テンプレートの拡充。
- **Phase 3: 収益化基盤の構築（リリース後半年以降）**
  - **目的**: ビジネスとしての持続可能性確保。
  - **内容**: 必要に応じたネイティブアプリ化の検討。

## リスク分析
1. **技術的リスク**
   - **内容**: Google Maps APIの利用料高騰や仕様変更。動画生成処理によるサーバー負荷増大。
   - **対策**: 初期はGoogleマイマップへのKMLエクスポートを中心としAPIコールを最小化。動画生成はRemotion等のクライアントサイドレンダリング技術を活用、または非同期キューによる負荷分散を行う。
2. **ビジネス的リスク**
   - **内容**: 初期コンテンツ（旅程データ）が不足し、検索してもヒットしない（コールドスタート問題）。
   - **対策**: 運営側やアンバサダー（友人・知人）によるシードデータの事前投入を数十件規模で行う。
3. **法的/コンプライアンスリスク**
   - **内容**: 子供の写真や位置情報などプライバシー性の高いデータの漏洩。
   - **対策**: デフォルトの公開範囲設定を慎重に行う（顔が写っている写真は限定公開・動画生成専用にする等）。利用規約およびプライバシーポリシーの整備。

## UI/UXの方向性
- **デザインコンセプト**: World Traveler（空港・パスポート・旅）。ワクワクする非日常感と、情報を探しやすいモダンさを両立。
- **トーン＆マナー**: ワクワク感、信頼感、思い出の温かみ。メインカラーは深いネイビーとゴールド、失敗アラートは視認性の高いイエロー。
- **ターゲットデバイス**: モバイルファースト（現地での利用や写真投稿を考慮）だが、計画作成画面はPC/タブレットの広い画面幅に最適化。
- **参考サイト/アプリ**:
  - Airbnb (検索のしやすさ、写真の見せ方)
  - Notion (ドラッグ＆ドロップなどの直感的な操作感、すっきりしたデザイン)
  - Flighty (航空券・空港をモチーフにした洗練されたUI)
